# AI Threat Hunting — Query Generation & Evaluation System

Translates natural-language threat-hunting hypotheses into executable SQL queries against the [flaws.cloud CloudTrail dataset](https://www.kaggle.com/datasets/flaws/cloudtrailflaws), then scores the generated queries against provided ground-truth outcomes using Precision / Recall / F1.

---

## Quick Start

```bash
# 1. Copy environment template and fill in your values
cp .env.example .env
# Required: OPENAI_API_KEY, LLM_MODEL (e.g. gpt-4o-mini)

# 2. Start PostgreSQL & the Web UI
docker compose up --build -d

# 3. Access the Interactive Web UI
# Open your browser and navigate to: http://localhost:8501
# The Streamlit UI provides:
# - Single Hypothesis: Step-by-step query generation and execution.
# - Complete Evaluation: One-click batch generation and scoring for all hypotheses.
# - Results: A historical browser for timestamped evaluation reports.
# - Approach & Iterations: A view to read through the project progression and metrics.

# 4. (Optional) Run via CLI
# Create a virtual environment, install dependencies, and run:
python3 -m venv .venv
source .venv/bin/activate
pip install -r requirements.txt
python -m src.main all
```

Outputs:
- UI runs are saved in `results/YYYYMMDD_HHMMSS/`
- CLI runs generate `generated_queries.json`, `evaluation_results.json`, and `EVALUATION_REPORT.md` in the `results/` folder.

## Design Decisions & Trade-offs

### SQL over Pandas / Polars
PostgreSQL handles 1.94M rows efficiently without loading the entire dataset into memory. Indexes on `eventName`, `eventTime`, `userIdentitytype`, `errorCode`, `userAgent`, and `sourceIPAddress` make the hypotheses-relevant filters fast. Pandas operations would have required the full 1 GB CSV in RAM on every run.

### Postgres over DuckDB
PostgreSQL provides a persistent, shared store — critical for realistic threat-hunting scenarios and for reproducible evaluation (the data is ingested once). DuckDB would have been lighter but less representative of production security tooling.

### JSON-mode LLM output over free-form parsing
Using `response_format={"type":"json_object"}` with a Pydantic schema ensures the LLM always returns a parseable structure. Free-form responses require fragile regex extraction that breaks across model versions.

### eventID-keyed scoring
CloudTrail rows have unique `eventID` values, making precision/recall computation straightforward. Hypothesis 1 (sign-in failures) is an exception — its ground truth was projected onto `(eventTime, sourceIPAddress, userIdentityuserName)`; we handle this with `HYP_KEY_MAP` in `config.py`.

### Schema-aware prompting
The system prompt includes the exact column names and types from the Postgres table. This dramatically reduces hallucinated column names — a common failure mode in text-to-SQL systems.

## Extending to Other Datasets

1. **Ingest**: Replace `data/nineteenFeaturesDf.csv` path in `.env` (`CSV_PATH`).  Update `SCHEMA_COLUMNS` in `src/config.py` to match your dataset's columns. The ingestion function (`src/db.py::ingest_csv`) uses the `COPY` protocol and is dataset-agnostic.

2. **Prompt schema hint**: `src/prompts.py::SYSTEM_PROMPT` reads column names from `config.SCHEMA_COLUMNS`. Adding a new dataset just requires updating `SCHEMA_COLUMNS`.

3. **Scoring keys**: Add entries to `HYP_KEY_MAP` in `src/config.py` if your ground truth uses non-standard ID columns.

4. **Ground truth format**: Must match the `hypotheses_outcomes.json` format (list of `{hyp_id: column→value-list}` objects). The `utils.py::load_hypotheses_outcomes` function handles parsing.
